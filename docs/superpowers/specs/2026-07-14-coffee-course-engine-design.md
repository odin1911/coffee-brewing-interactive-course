# 数据驱动的交互式课件引擎设计

日期：2026-07-14

## 1. 背景

本项目需要在 4 小时内交付一个可在浏览器放映、由外部 JSON 驱动、具有互动能力并能导出完整 PDF 的课件引擎。示例课程是《咖啡冲煮入门》，但评审会通过替换 JSON 验证引擎能否承载完全不同的课程。

内容与素材由 AI 生成或从公开网页收集。运行时不请求 AI、第三方内容接口或远程非图片素材；`course.json` 可配置公开 HTTPS 图片。

## 2. 目标与非目标

### 目标

- 只修改 `course.json` 即可更换课程、品牌、页面顺序、图表数据与分支；新图片可使用 HTTPS 或 base64 raster data URL。
- 实现答题即时统计、图表点击钻取和答题分支三类交互。
- 网页放映时只经过一个分支；PDF 包含两个分支和全部钻取明细。
- 本地运行时持续保存每次学习会话，用于分析路径、有效时长和中途流失位置。
- 提供清晰的一键启动、构建、测试和 PDF 导出说明。
- 保持代码量和依赖量足够小，便于 20 分钟现场功能修改和答辩解释。

### 非目标

- 不实现多设备投票、账号、身份识别、数据库、云端同步或实时后台。
- 不在运行时抓取网页或调用生成式 AI。
- 不实现 PPTX 导出。
- 不构建通用低代码页面编辑器或插件系统。
- 不把导出、记录状态或开发工具混入教学页面。

## 3. 已批准的产品决策

- 技术栈：React、TypeScript、Vite。
- 视觉方向：温暖编辑感，使用奶油底、咖啡棕和琥珀橙。
- 答题方式：单浏览器，答题页穿插在教学流程中。
- 答题后先展示统计，由讲师点击“继续”进入选项对应的分支。
- PDF 通过同一套 HTML 和打印样式生成，包含完整课程、两个分支和全部图表明细。
- 内容配置固化到 `course.json`，素材可使用本地文件、HTTPS 或 base64 raster data URL，来源记录在 `sources.json`。
- 本地 `reference-design-contract` skill 只生成 `DESIGN.md` 和参考页面，不作为项目运行依赖。
- 本地服务持续更新每次学习会话对应的独立 JSON 文件；静态部署不启用记录功能。
- 第 8 页只显示 JSON 定义的教学总结与 CTA。PDF、记录状态、重新开始和校验结果位于独立工具页。

## 4. 设计准备安排

正式 4 小时考核前，使用本地 `reference-design-contract`、`frontend-design` 和 `imagegen` 完成设计输入：

1. 生成温暖编辑风格的 `DESIGN.md`。
2. 生成封面、图表页、答题页三张 16:9 HTML 参考页面。
3. 生成本地封面图与品牌图形，并检查浅色容器中的文本对比度。

这些工具只提供文件和设计约束，不接入额外服务或项目运行时。

## 5. 系统架构

系统分为三个路由：

- `/course`：教学放映。记录教学路径、页面有效时长和课件交互。
- `/tools`：独立工具页。提供完整 PDF 导出、记录状态、重新开始和配置校验结果，不计入教学路径或学习时长。
- `/print`：打印专用页面。渲染全部页面定义、两个分支和静态展开的图表详情，然后调用浏览器打印。

引入 `react-router-dom` 管理三个清晰的页面边界。课程会话状态位于路由之上，因此在 `/course` 与 `/tools` 之间切换不会丢失。

本地开发由 `server.mjs` 提供单一入口：Express 承载记录 API，并在开发模式接入 Vite 中间件。静态构建只输出前端文件；前端探测不到本地记录 API 时自动关闭记录能力。

## 6. 组件职责

- `CourseLoader`：在运行时读取 `course.json`，完成结构与引用校验。
- `PresentationShell`：提供 16:9 舞台、进度、全屏、键盘和按钮导航。
- `SlideRenderer`：根据 `slide.type` 分发五种明确页面组件。
- `CoverSlide`：标题、主讲人、品牌标识和可选封面图。
- `ContentSlide`：标题、要点和可选图片。
- `ChartSlide`：原生 SVG 柱状图、选择状态和明细面板。
- `QuizSlide`：选项、当前统计和“继续”操作。
- `CtaSlide`：仅显示课程总结、正文和行动号召。
- `CourseSession`：用 `useReducer` 管理当前页面、实际路径、答案、统计与钻取状态。
- `SessionRecorder`：结算有效时长、生成会话快照并调用本地 API。
- `ToolsPage`：导出、记录状态、重新开始和校验结果。
- `PrintDeck`：复用页面组件，以打印模式渲染完整课程。

本次不引入独立状态库或图表库。一个顶层 reducer 足以覆盖单个课程会话；两张柱状图由一个 SVG 组件完成，打印稳定且点击逻辑直接。若以后出现跨课程共享状态或多种复杂图表，再分别评估 Zustand 和 ECharts。

## 7. 课程数据契约

`course.json` 保留考题样例的所有语义，并增加课程识别、主题、界面文案、详情和明确汇合路径。

```json
{
  "course": {
    "id": "coffee-brewing-intro",
    "version": "1.0.0",
    "title": "咖啡冲煮入门",
    "presenter": "主讲人姓名",
    "brand": {
      "primary": "#4A2C1A",
      "accent": "#D89A4E",
      "background": "#F7F0E5",
      "text": "#2B211B",
      "surface": "#FFF9F0",
      "muted": "#6B5545",
      "line": "#D8C7B5",
      "chartColors": ["#A95532", "#D89A4E", "#65704B"],
      "logo": "assets/logo.png"
    }
  },
  "ui": {
    "previous": "上一页",
    "next": "下一页",
    "continue": "继续",
    "showDetail": "查看明细",
    "closeDetail": "关闭明细",
    "exportPdf": "导出完整 PDF",
    "openTools": "扩展工具",
    "recordStatus": "学习记录状态",
    "restartCourse": "重新开始课程",
    "validationResult": "配置校验结果",
    "backToCourse": "返回课程",
    "select": "记一票",
    "selected": "当前领先",
    "results": "现场计票",
    "peopleUnit": "人",
    "imageLoadError": "图片加载失败，无法导出"
  },
  "slides": [],
  "details": {}
}
```

页面类型：

- 所有页面可选 `kicker`；默认按数组顺序前进，`next` 用于显式跳转。
- `cover`：`title`，以及可选 `subtitle`、`topics`、`image`、`imageAlt`。
- `content`：`title`、`bullets`、可选 `image` 和 `imageAlt`。
- `chart`：`title`、`chart.kind`、`unit`、`clickable`、`series[]`。
- `quiz`：`question`、可选 `description`、`options[]`；每个选项包含 `id`、`text` 和 `goto`。
- `cta`：`title`、`body` 和 `action`；`action` 包含 `label` 与本地或公开 `href`，不包含引擎工具。

页面默认按数组顺序前进。需要跳转或汇合的页面使用显式 `next`。每个图表数据项通过 `detail` 引用顶层 `details`。

所有可见课程文字、界面标签、图片地址、图片替代文字、品牌色、页面顺序、图表数据和 CTA 行为都来自 JSON。图片允许站内路径、HTTPS 或 base64 raster data URL；JSON 不允许嵌入任意 HTML 或脚本。

`sources.json` 为每条课程事实和素材保存来源类型。网页来源记录标题、URL、访问日期和用途；AI 生成内容记录工具、提示词摘要、生成日期和用途。该文件只用于审计，不在运行时加载。

## 8. 教学与交互流程

考题大纲包含“手冲咖啡基本步骤”，但最小 JSON 样例遗漏了该对象。设计以大纲为准补齐该页。因此配置包含九个页面定义，而任一实际学习路径仍为八页：

1. 封面。
2. 三大产区与风味。
3. 咖啡因含量图表与钻取。
4. 手冲咖啡基本步骤。
5. 互动答题与即时统计。
6. 入门建议或进阶建议，两者只进入其一。
7. 不同冲煮方式萃取率与钻取。
8. 教学总结与 CTA。

答题选项从 0 票开始，由主讲人根据听众回答重复加票并减票纠错，页面即时显示现场票数和百分比。唯一最高票选项按 `goto` 进入分支；零票或平票时不能继续。后退使用实际 `visitedPath`，返回答题页修正票数后可进入新的最高票分支。

图表柱形是可聚焦按钮。点击后展开对应详情并开始记录功能使用时长；关闭详情或离开页面时结束计时。打印模式不需要点击，直接展示全部详情。

## 9. PDF 与工具页

`/tools` 不属于课件内容，也不出现在页面计数、学习路径或 PDF 中。进入工具页时结束当前教学片段并暂停学习时长。

工具页提供：

- 导出完整 PDF。
- 显示本地记录是否启用、当前会话 ID 和最近保存时间。
- 重新开始课程并创建新会话。
- 显示课程配置与素材校验结果。

`/print` 使用同一批页面组件和单独的打印样式。它渲染九个页面定义，明确标注分支 A 与分支 B，并静态展开所有图表明细。页面尺寸按 16:9 横向设计；工具页调用 `window.print()` 完成 HTML 到 PDF 的转换。

## 10. 学习记录

本地运行开始课程时调用 `POST /api/sessions`。服务端生成 UUID、创建 `records/<session-id>.json`，并返回会话 ID。客户端不提供文件路径。

“一次学习会话”定义为一次进入 `/course` 后开始的独立运行。重新加载 `/course` 或在工具页点击“重新开始课程”都会创建新文件；旧文件保持 `in_progress`，从而保留中途退出证据。

客户端在以下时机调用 `POST /api/sessions/:id` 更新完整快照：

- 完成或离开一页。
- 答题、继续分支、打开或关闭图表详情。
- 页面进入后台或恢复可见。
- 进入工具页。
- 正常完成课程。

`pagehide` 使用 `navigator.sendBeacon` 尽力提交最后快照。客户端只允许一个更新请求在途；若期间产生新快照，只保留最新一个。失败的最新快照在下一个记录时机重试。

服务端先把快照写入同目录临时文件，再通过 `rename` 原子替换会话文件。更新接口只接受服务端生成且格式合法的 UUID，不能由请求参数构造任意文件路径。

记录 API 将请求体限制为 1 MB，并校验 `schemaVersion`、会话 ID、时间字段和数组结构。客户端提交的 `sessionId` 必须与 URL 中的服务端会话 ID 一致。

记录结构：

```json
{
  "schemaVersion": 1,
  "sessionId": "uuid",
  "courseId": "coffee-brewing-intro",
  "courseVersion": "1.0.0",
  "status": "in_progress",
  "startedAt": "ISO-8601",
  "updatedAt": "ISO-8601",
  "completedAt": null,
  "lastSlideId": "quiz1",
  "visitedPath": ["cover", "beans", "caffeine", "steps", "quiz1"],
  "totalActiveMs": 0,
  "slideSegments": [],
  "featureEvents": []
}
```

页面隐藏、工具页停留和浏览器后台挂起不计入学习时长。记录中不包含姓名、邮箱、设备指纹或其他个人身份数据。

## 11. 视觉与无障碍

`DESIGN.md` 固定温暖浅色主题，不跟随操作系统深色模式改变课件前景色。普通文本与背景对比度至少为 4.5:1，大号文本至少为 3:1。任何浅色容器必须同时声明明确的深色前景色，不能依赖外层继承。

主要规则：

- 16:9 舞台，投影时保证标题、正文和图表标签可读。
- 奶油色作为主背景，咖啡棕用于标题和主要操作，琥珀橙只作强调。
- 所有交互支持键盘操作，并提供可见焦点。
- 图表柱形使用按钮语义和可读标签，信息不只依赖颜色。
- 图片替代文字来自 JSON。
- 减少动态效果设置开启时禁用非必要转场。
- 打印样式显式重置背景、文字色、阴影和分页规则。

## 12. 错误处理

- JSON 无法解析、页面 ID 重复、无效 `goto`、`next` 或 `detail` 时阻止放映，并显示字段路径和修复提示。
- 素材缺失时显示保持版式的占位，并在工具页汇总错误。
- 本地记录 API 不可用时，教学与 PDF 功能继续工作；静态部署不显示记录操作。
- 记录更新失败时显示在工具页，并按下一次记录时机重试最新快照。
- 打印模式发现配置错误时不调用打印，先展示校验结果。

## 13. 验证策略

使用 Vitest 覆盖最有风险的纯逻辑：

- 课程校验拒绝重复 ID、无效分支目标和无效详情引用。
- reducer 能正常前进、按答案分支、按实际路径后退，并在修改答案时截断旧路径。
- 会话计时在页面隐藏期间不累计。
- 会话快照正确记录最后页面、有效时长和功能事件。

本地 API 验证：

- 每次开始课程生成不同文件。
- 更新只写入当前会话文件。
- 非法会话 ID 不能访问文件系统路径。
- 临时写入失败时不破坏已有 JSON。

浏览器冒烟验证：

- 方向键、按钮、全屏和返回行为。
- 两张图表的钻取与关闭。
- 答题统计、继续和两个分支。
- 从课程进入工具页后学习计时暂停。
- `/print` 包含两个分支和全部明细，不包含工具页。
- 用另一门完全不同的课程 JSON 替换示例配置，代码零修改仍可放映。

最终门槛：`npm test`、`npm run build`、浏览器完整走查和 PDF 逐页检查全部通过。

## 15. 四小时时间预算

设计文件已在考核前生成，正式计时内不再配置设计工具。

1. 0–20 分钟：项目初始化、更新 `Contract.md`、课程 JSON 与设计令牌。
2. 20–90 分钟：运行时加载、五类页面、导航与路由。
3. 90–145 分钟：答题、统计、分支与图表钻取。
4. 145–180 分钟：持续记录、本地 API 与工具页。
5. 180–210 分钟：温暖编辑视觉、响应式和打印模式。
